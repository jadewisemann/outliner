# Local-First Outliner MVP 개발 명세서

Dynalist의 핵심 경험인 무한 뎁스 아웃라이너를 로컬 퍼스트 웹 앱으로 구현한다. MVP의 목표는 "항상 즉시 열리고, 즉시 입력되고, 키보드만으로 빠르게 구조화되며, 개인의 여러 기기에서 안전하게 병합되는 단일 아웃라인 문서"다.

상세 개발 문서는 [docs/README.md](./docs/README.md)를 진입점으로 사용한다. 요구사항, 개발 단계, 테스트 전략, 아키텍처, 개발 규칙, 결정 기록은 `docs/` 아래 문서를 기준으로 최신 상태를 유지한다.

## 1. 제품 범위

### 1.1 핵심 목표

- 단일 워크스페이스: 사용자는 하나의 영구 Root 문서만 가진다.
- 무한 뎁스 아웃라이너: 각 줄은 노드이며, 노드는 텍스트와 하위 노드를 가진다.
- 로컬 우선: 앱 실행, 읽기, 편집은 네트워크 상태와 무관하게 동작한다.
- 충돌 없는 개인 다기기 동기화: 같은 사용자의 여러 기기 동시 편집은 CRDT로 병합한다.
- 빠른 조작감: 일반 타이핑, 들여쓰기, 접기/펼치기, 줌인은 즉시 반응해야 한다.
- 키보드 우선: 핵심 조작은 마우스 없이 수행 가능해야 한다.

### 1.2 MVP에서 제외

- 파일/폴더, 다중 문서, 문서 공유
- 종단간 암호화(E2EE)
- 이미지/첨부파일
- 복잡한 권한 모델
- 서버 사이드 비즈니스 로직
- 공개 링크, 팀 협업, 댓글
- 전체 텍스트 검색 인덱싱 고도화
- 모바일 앱 패키징은 웹 MVP 이후로 분리

### 1.3 성공 기준

- 네트워크 오프라인 상태에서 앱을 새로 열어도 마지막 로컬 상태가 1초 이내 표시된다.
- 10,000개 노드를 가진 문서에서 기본 스크롤과 커서 이동이 눈에 띄게 끊기지 않는다.
- 구조적으로는 50,000개 노드까지 확장 가능하도록 설계한다.
- 두 브라우저 창에서 같은 노드를 동시에 편집해도 텍스트 유실 없이 병합된다.
- 새로고침 후에도 문서 내용, 접힘 상태, 현재 줌 위치가 복원된다.

## 2. 사용자 경험 명세

### 2.1 화면 구조

- 상단 바
  - 브레드크럼: `Root / Parent / Current`
  - 동기화 상태: `Saved locally`, `Syncing`, `Offline`, `Synced`, `Sync error`
  - Undo/Redo 버튼
- 본문
  - 현재 줌인된 노드의 자식 노드 목록을 편집 가능한 아웃라인으로 표시한다.
  - Root에 줌인한 상태가 기본 진입 화면이다.
- 하단/보조 상태
  - 모바일에서는 키보드 표시 상태에서 들여쓰기/내어쓰기/접기 버튼을 제공할 수 있다.

### 2.2 노드 모델

각 아웃라인 노드는 다음 속성을 가진다.

- `id`: 안정적인 고유 ID. Yjs map key 또는 Lexical node key와 별도로 앱 레벨 ID를 둔다.
- `text`: MVP에서는 플레인 텍스트 본문. 리치텍스트는 속도와 키보드 UX를 해치지 않는 범위에서 후속 단계로 둔다.
- `children`: 하위 노드 순서 목록.
- `collapsed`: 하위 노드 접힘 여부.
- `createdAt`: 최초 생성 시각.
- `updatedAt`: 마지막 수정 시각.

결정 사항:

- 노드 하나는 화면의 한 줄이다.
- 빈 노드도 유효한 노드다.
- Root 노드는 삭제할 수 없고 텍스트를 직접 편집하지 않는다.
- 형제 노드 순서는 CRDT 배열로 관리한다.

### 2.3 키보드 동작

- `Enter`
  - 현재 노드 뒤에 새 형제 노드를 만든다.
  - 커서가 노드 텍스트 중간에 있으면 현재 텍스트를 커서 기준으로 분리한다.
  - 현재 노드가 비어 있고 하위 노드가 없으면 일반 새 형제 노드 생성으로 처리한다.
- `Tab`
  - 현재 노드를 바로 위 형제 노드의 마지막 자식으로 이동한다.
  - 위 형제 노드가 없으면 아무 동작도 하지 않는다.
- `Shift+Tab`
  - 현재 노드를 부모의 다음 형제로 승격한다.
  - Root 직속 노드에서는 아무 동작도 하지 않는다.
- `Backspace`
  - 텍스트가 있으면 일반 삭제.
  - 빈 노드에서 누르면 이전 노드와 병합하거나 삭제한다.
  - 자식이 있는 빈 노드에서 누르면 현재 빈 노드는 제거하고 자식 노드는 같은 레벨로 승격한다. 자식 데이터는 삭제하지 않는다.
- `ArrowUp` / `ArrowDown`
  - 현재 보이는 노드 기준으로 이전/다음 노드로 이동한다.
  - 접힌 하위 노드는 이동 대상에서 제외한다.
- `Cmd/Ctrl+[` / `Cmd/Ctrl+]`
  - 내어쓰기/들여쓰기 단축키로도 지원한다.
- `Cmd/Ctrl+Z`, `Cmd/Ctrl+Shift+Z`
  - Yjs UndoManager 기반 Undo/Redo.

### 2.4 마우스/터치 동작

- 노드 bullet 클릭
  - 해당 노드로 줌인한다.
- 노드 왼쪽 토글 클릭
  - 하위 노드가 있으면 접기/펼치기.
  - 하위 노드가 없으면 토글을 표시하지 않거나 비활성으로 표시한다.
- 브레드크럼 클릭
  - 해당 상위 노드로 줌아웃한다.
- 드래그 앤 드롭
  - MVP 필수는 아니며 Phase 2 이후로 미룬다.

### 2.5 마크다운 단축키

MVP에서는 마크다운을 구조 변환의 핵심 기능으로 두지 않는다. Dynalist 대안으로서 가장 중요한 것은 빠른 노드 작성과 키보드 기반 구조 편집이다.

- `- `, `* `는 새 bullet list를 만들지 않고 현재 아웃라이너 노드의 텍스트로 유지한다.
- `# `, `**text**`, `` `text` ``, `[label](url)` 등 리치텍스트 변환은 Phase 3 이후 옵션으로 둔다.
- TODO/checkbox 노드는 MVP에서 제외한다.

## 3. 기술 아키텍처

### 3.1 기본 스택

- App: React + Vite + TypeScript
- Editor: Lexical
- CRDT: Yjs + `@lexical/yjs`
- Web local persistence: `y-indexeddb`
- Mobile shell: Capacitor, 웹 MVP 이후
- Mobile local persistence: 웹 MVP 이후 IndexedDB 유지 또는 SQLite 전환 검증
- Remote sync: Firebase Realtime Database
- Test: Vitest, React Testing Library, Playwright

### 3.2 핵심 설계 원칙

- Lexical은 편집 UI와 AST를 담당한다.
- Yjs는 동시성, 병합, undo/redo, 원격 업데이트 단위를 담당한다.
- 서버는 인증과 업데이트 릴레이만 담당한다.
- 클라이언트는 로컬 DB에서 먼저 복원하고, 이후 원격 업데이트를 병합한다.
- 원격 저장소는 문서 전체 덮어쓰기만으로 동기화하지 않는다. 동시 업데이트 유실을 피하기 위해 업데이트 로그와 스냅샷을 분리한다.
- 모든 기능은 TDD로 개발한다. 실패하는 테스트를 먼저 작성하고, 최소 구현으로 통과시킨 뒤, 동작을 보존한 상태에서 리팩터링한다.

### 3.3 Yjs 문서 구조 초안

```ts
type WorkspaceYDoc = {
  lexical: XmlText | XmlFragment;
  nodeMeta: Y.Map<NodeMeta>;
  appState: Y.Map<AppState>;
};

type NodeMeta = {
  id: string;
  collapsed: boolean;
  createdAt: number;
  updatedAt: number;
};

type AppState = {
  zoomNodeId: string;
  selectionNodeId?: string;
};
```

구현 중 `@lexical/yjs`가 요구하는 실제 바인딩 구조에 맞춰 `lexical` 필드 타입은 조정한다.

### 3.4 원격 동기화 구조

Firebase 경로 초안:

```txt
users/{userId}/workspaces/root/
  snapshot/
    state: string        # Base64 encoded Y.encodeStateAsUpdate(doc)
    vector: string       # Base64 encoded Y.encodeStateVector(doc)
    updatedAt: number
  updates/{updateId}/
    clientId: string
    seq: number
    update: string       # Base64 encoded Yjs update
    createdAt: number
```

동작:

1. 앱 시작 시 IndexedDB/SQLite에서 Y.Doc를 즉시 복원한다.
2. 인증과 네트워크가 가능하면 원격 snapshot을 가져와 `Y.applyUpdate` 한다.
3. 원격 updates 중 아직 적용하지 않은 update를 순서와 무관하게 적용한다.
4. 로컬 변경은 debounce 후 updates에 append한다.
5. 일정 업데이트 수 또는 용량을 넘으면 snapshot을 갱신하고 오래된 updates를 정리한다.

초기 MVP에서는 정리(compaction)를 수동 또는 단순 기준으로 처리한다.

### 3.5 오프라인 큐

- 로컬 변경은 즉시 Y.Doc와 로컬 DB에 반영된다.
- 네트워크가 없으면 원격 전송 대기 큐에 남긴다.
- 재연결 시 누락 update를 전송한다.
- 같은 update가 중복 수신되어도 Yjs 적용은 idempotent하게 처리되어야 한다.

## 4. 저장소 및 데이터 보존

### 4.1 웹 로컬 저장소

- `y-indexeddb` provider를 사용한다.
- 문서 이름은 `workspace_root`로 고정한다.
- 앱 UI 상태 중 CRDT에 넣지 않을 값은 별도 localStorage 또는 IndexedDB key에 저장한다.
  - 예: 마지막 스크롤 위치, 사이드바 열림 여부

### 4.2 모바일 로컬 저장소

- 웹 MVP에서는 모바일 앱 패키징을 제외한다.
- 모바일 단계에서는 Capacitor 환경 감지 후 IndexedDB 유지 또는 SQLite 기반 persistence 전환을 검증한다.
- `y-sqlite`가 웹뷰/Capacitor 환경에서 안정적으로 동작하는지 검증 필요.
- 검증이 어렵다면 모바일 첫 버전은 IndexedDB를 유지하고 네이티브 SQLite 전환을 별도 Phase로 분리한다.

### 4.3 데이터 내보내기/복구

MVP에 최소한의 안전장치를 둔다.

- JSON 내보내기: 현재 아웃라인 트리를 사람이 읽을 수 있는 JSON으로 export
- Markdown 내보내기: 들여쓰기 기반 Markdown export
- Import는 MVP 후순위

## 5. TDD 개발 원칙

### 5.1 기본 루프

모든 기능은 다음 순서로 진행한다.

1. Red: 기대 동작을 테스트로 먼저 작성하고 실패를 확인한다.
2. Green: 테스트를 통과시키는 최소 구현을 작성한다.
3. Refactor: 테스트를 유지한 채 중복, 이름, 책임 경계를 정리한다.
4. Commit-ready: 관련 단위/통합 테스트와 타입 체크가 통과한 상태에서만 작업을 완료로 본다.

### 5.2 테스트 우선순위

- 순수 도메인 로직은 반드시 단위 테스트를 먼저 작성한다.
  - 예: 노드 생성, 분리, 병합, 들여쓰기, 내어쓰기, 접기 상태, visible nodes 계산
- 외부 라이브러리와 연결되는 부분은 얇은 adapter를 두고 통합 테스트로 검증한다.
  - 예: Lexical command adapter, Yjs document adapter, persistence provider
- 사용자 핵심 흐름은 Playwright E2E 테스트로 검증한다.
  - 예: 노드 작성, 들여쓰기, 접기, 줌인, 새로고침 복원
- 시각적 세부나 라이브러리 내부 동작은 직접 테스트하지 않는다.

### 5.3 테스트 가능한 설계 규칙

- 트리 조작 로직은 React 컴포넌트 밖의 순수 함수로 작성한다.
- Lexical/Yjs/Firebase 직접 호출은 adapter 계층에 격리한다.
- UI 컴포넌트는 상태를 직접 변형하지 않고 command 함수를 호출한다.
- 시간, ID 생성, 네트워크 상태는 테스트에서 주입 가능하게 만든다.
- 동기화 로직은 실제 Firebase 없이 fake remote store로 먼저 검증한다.

### 5.4 테스트 파일 배치

```txt
src/
  domain/
    outline.ts
    outline.test.ts
  editor/
    lexicalAdapter.ts
    lexicalAdapter.test.ts
  sync/
    yjsAdapter.ts
    yjsAdapter.test.ts
    remoteSync.ts
    remoteSync.test.ts
  persistence/
    localPersistence.ts
    localPersistence.test.ts
  components/
    Outliner.tsx
    Outliner.test.tsx
e2e/
  outliner.spec.ts
  persistence.spec.ts
  sync.spec.ts
```

### 5.5 Definition of Done

기능 하나는 아래 조건을 만족해야 완료로 본다.

- 실패하는 테스트가 먼저 존재했다.
- 구현 후 관련 테스트가 통과한다.
- 리팩터링 후에도 테스트가 통과한다.
- 새로 추가된 핵심 로직은 최소 하나 이상의 경계 조건 테스트를 가진다.
- 버그 수정은 재현 테스트를 먼저 추가한다.
- 테스트 이름이 제품 동작을 설명한다.

## 6. 구현 계획

### Phase 0: 프로젝트 부트스트랩

- Vite + React + TypeScript 설정
- ESLint/Prettier 또는 Biome 설정
- Vitest, Playwright 설치
- 기본 앱 레이아웃 작성
- Firebase 설정값 주입 방식 결정. 단, Phase 0~2는 로그인과 원격 설정 없이 로컬 앱으로 동작해야 한다.
- TDD용 테스트 스크립트 구성: `test`, `test:watch`, `test:e2e`, `typecheck`

완료 기준:

- `npm run dev`로 웹 앱 실행
- `npm run test` 실행 가능
- `npm run test:watch`로 Red-Green-Refactor 루프 실행 가능
- Playwright 기본 smoke test 실행 가능
- 빈 Root 아웃라인 화면 표시

### Phase 1: 로컬 아웃라이너 편집

- `outline` 도메인 모델 단위 테스트 작성
- Lexical 에디터 마운트
- Enter/Tab/Shift+Tab/Backspace 동작 테스트 작성
- 한 줄 하나의 노드처럼 보이는 기본 렌더링
- Enter로 노드 생성/분리
- Tab/Shift+Tab으로 계층 이동
- 접기/펼치기 상태 구현
- 브레드크럼과 줌인/줌아웃 구현
- 마크다운 자동 변환은 연결하지 않는다. `- `, `* `, `# ` 등은 플레인 텍스트 입력으로 유지한다.
- 핵심 편집 흐름 E2E 테스트 작성

완료 기준:

- 트리 조작 단위 테스트 통과
- 편집 command 통합 테스트 통과
- 키보드만으로 3뎁스 이상 아웃라인 작성 가능
- 접힌 노드의 자식은 화면과 커서 이동 대상에서 제외
- 새로고침 전까지 기본 편집 UX가 안정적으로 동작

### Phase 2: 로컬 퍼시스턴스와 Undo/Redo

- Y.Doc adapter 테스트 작성
- Y.Doc 초기화
- Lexical-Yjs binding 적용
- persistence 복원 테스트 작성

## 8. 주요 리스크와 대응

- Lexical 기본 리스트 모델과 아웃라이너 모델이 충돌할 수 있다.
  - 대응: 기본 ListNode를 그대로 쓰기 전에 커스텀 노드/플러그인 구조를 짧게 프로토타이핑한다.
- `@lexical/yjs`와 커스텀 노드 메타데이터 동기화가 복잡할 수 있다.
  - 대응: 텍스트 AST와 노드 메타데이터를 분리하고, ID 매핑 규칙을 먼저 확정한다.
- Firebase RTDB에 바이너리 업데이트를 append하는 방식은 저장소가 커질 수 있다.
  - 대응: snapshot + updates 구조를 사용하고 compaction 기준을 둔다.
- 모바일 SQLite provider 호환성이 불확실하다.
  - 대응: 모바일 persistence는 별도 검증 마일스톤으로 분리한다.
- 대량 노드에서 Lexical 전체 에디터 가상화가 까다로울 수 있다.
  - 대응: 초기에는 줌인/접힘으로 렌더링 범위를 줄이고, Phase 4에서 본격 가상화를 적용한다.
- TDD가 UI/CRDT 라이브러리 통합부에서 느려질 수 있다.
  - 대응: 순수 도메인 로직, adapter 계약 테스트, 소수의 E2E로 층을 나누고 모든 케이스를 브라우저 테스트에 몰지 않는다.

## 9. 제품 결정

Dynalist 대안을 빠르게 만들기 위해 다음 결정을 기본값으로 확정한다.

1. Phase 0~2는 로그인 없는 로컬 단일 문서로 시작한다. 원격 동기화는 이후 개인 다기기 동기화로 추가한다.
2. 원격 동기화 저장소는 Firebase Realtime Database를 사용한다.
3. MVP 텍스트는 플레인 텍스트 중심이다. 리치텍스트와 마크다운 변환은 키보드 속도를 해치지 않을 때 후속 단계로 추가한다.
4. 체크박스/TODO 노드는 MVP에서 제외한다.
5. 자식이 있는 빈 노드에서 `Backspace`를 누르면 빈 부모를 제거하고 자식들을 같은 레벨로 승격한다.
6. bullet 클릭은 Dynalist처럼 줌인이다. 드래그 핸들은 후속 기능이다.
7. 모바일 앱은 웹 MVP 검증 후 별도 단계로 진행한다.
8. 협업 범위는 개인 다기기 동기화다. 여러 사용자 실시간 공동편집과 공유 권한은 제외한다.
9. 내보내기는 JSON과 Markdown을 모두 지원한다. JSON은 복구용, Markdown은 이동성/호환성용이다.
10. MVP 성능 수용 기준은 10,000개 노드이며, 구조 설계 목표는 50,000개 노드다.
11. TDD 규칙은 CI에서 `typecheck`, 단위 테스트, 핵심 smoke E2E로 강제한다.
